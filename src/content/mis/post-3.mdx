# Vocab 

This section is all about going from a business problem to a good database design. 

Why do we care to make a good database design?

It saves us money. 

There are companies will millions of data entries everyday and if that data is not properly organized or there is a bunch of repeated data it can cost us. 

# Entities and Attributes

Entities are places, persons or things we want to store information about. 

In other words, they are your table names. 

Attributes are well, the attributes of these entities. 

In other words, the specific data you want to collect about that entitie. 

Also, in other words, the names of your columns in your tables. 

For example, if we want to collect data about students, specifically we want to collect their first name, last name, netID and major. 

Here, 

Entitie(s): Student 
Attributes: first name, last name,netID, major

# Primary keys

A unique identifier for a table. A column that will not repeat data entires. 

For example, here, the primary key is the netID. As Rutgers students, we all have our own netID that nobody else does. Hence, we can use it to identify students. 

Why wouldn't we use names as a unique identifier? Because different people can have the same names like the Liams so it's data that can be repeated hence not fit to be the primary key. Same thing with majors. Thousands of students have the same majors. So it would be hard to identify a student by just their major.

So because there is more than one primary key it's called a composite key.

## Functional Dependencies

There are two types of functional dependencies.

### Transitive Dependency

When one attribute is dependent on another attribute other than the PK.

### Partial Dependency (Only occurs when you have a Composite Key)

When one attribute is dependent on only **part** of the PK. 


## Anomolies

Why don't we like dependencies? Because they create anomolies in our data. 

What are the types of anomolies? (**DUI**)

**1.** Deletion  
**2.** Updation   
**3.** Insertion    

Let's explain what they are through an example.

**1. Deletion:** What if a member wants to leave a course? Then the whole row, the member and it's corresponding course and fee will have to be deleted too and now we won't have a record of that course or it's fee. 

**2. Updation:** If we update price for the Yoga basics course from $50 to $60 we would have to do it for every member, for every entry which can lead to updation anamolies.

**3. Insertion:** What if we want to insert a new course but no one has signed up for it yet? This creates an insertion anmoly because you would have to create a dummy variable for member so you're able to actually have the course name visible.


Hence, the goal for database design is to remove data redundency and dependencies. 

How do we do that?

We normalize the database design. 

Normalization is the process of removing data redundency and dependencies. 

It's really a fancy way of just saying make more tables.
